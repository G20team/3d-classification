# Data Pipeline

The repository does not track Pokemon assets, generated mesh caches, render caches, or training outputs.
Users recreate them locally from upstream sources.

## 1. Fetch Assets

```bash
uv run python scripts/fetch_assets.py --output data/raw_assets
```

This command performs a shallow clone of the upstream Pokemon 3D assets repository into `data/raw_assets`.
The code does not assume a fixed internal directory layout and discovers `*.glb` files recursively.

## 2. Audit Assets

```bash
uv run python scripts/audit_assets.py \
  --asset-root data/raw_assets \
  --output data/manifests/asset_audit.jsonl
```

The audit step identifies usable regular-form assets and writes manifests such as:

```text
data/manifests/asset_audit.jsonl
data/manifests/selected_regular.jsonl
data/manifests/asset_audit_summary.md
```

Review `asset_audit_summary.md` before training. A large number of missing IDs or load failures usually
indicates an asset path, dependency, or upstream layout issue.

## 3. Visual Inspection

```bash
uv run python scripts/render_contact_sheet.py \
  --manifest data/manifests/selected_regular.jsonl \
  --output outputs/asset_audit_contact_sheet.png
```

Use the contact sheet to catch obviously broken meshes, wrong forms, or assets that should be excluded
from the closed-set catalog.

## 4. Mesh Cache

```bash
uv run python scripts/prepare_mesh_cache.py \
  --manifest data/manifests/selected_regular.jsonl \
  --output-root data/mesh_cache
```

The mesh cache normalizes geometry once so each experiment reads the same processed meshes. The cache step
merges mesh geometry, removes invalid geometry where possible, recenters each mesh, and scales it to a
consistent unit-sphere-like range.

## 5. Pose Splits

```bash
uv run python scripts/build_splits.py --config configs/splits_stratified.yaml
uv run python scripts/validate_splits.py --splits data/manifests/pose_splits_stratified_seed0.json
```

The standard split assigns each Pokemon-pose pair once, with 9/4/4 samples per class in train,
validation, and test. Every split contains all 17 pose conditions, and the per-pose counts differ by at
most two samples. The legacy condition-exclusive split remains available for old checkpoint reproduction.

## Checks Before Training

- `data/manifests/selected_regular.jsonl` contains the expected catalog.
- `data/mesh_cache` exists and can be inspected for a few representative assets.
- `pose_splits_stratified_seed0.json` passes class-quota, pose-balance, digest, and overlap validation.
- The same manifests and splits are used for all four main conditions.

## 6. Fixed-view RGB Render Cache

Single-view, Fixed Ring-4, and View Transformer reuse lossless PNG strips generated by the same PyTorch3D
renderer used by online mesh training. Build the one-view and shared four-view caches before full runs:

```bash
uv run python scripts/prepare_rgb_render_cache.py --config configs/single_view.yaml
uv run python scripts/prepare_rgb_render_cache.py --config configs/fixed_ring4.yaml
```

The cache identity includes SHA-256 digests of the source manifest and split plus the camera and renderer
settings. Fixed Ring-4 and View Transformer therefore share one cache. MVTN must keep online differentiable
rendering because its camera offsets change during training.

After data preparation, continue with the [experiment design overview](experiments/index.md) and run the
debug subset first.
